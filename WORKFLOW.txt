1. PROJECT STRUCTURE

BACKEND  (runs on http://localhost:8000)

      order_item.py   → OrderItemCreate, OrderItemOut
      product.py      → ProductCreate, ProductUpdate, ProductOut

    routes/           → FastAPI route handlers (API endpoints)
      auth.py         → /auth/register, /auth/login
                        also exports get_current_user() dependency
      orders.py       → /orders/ CRUD + status update + payment
      kitchen.py      → /kitchen/orders, /kitchen/dashboard
      products.py     → /products/ CRUD
      tables.py       → /tables/ CRUD
      analytics.py    → /analytics/overview, /daily-revenue, /top-products, /staff

    services/

    websockets/

                        - order_connections[]   → for /ws/orders (KDS + CDS)

                        - broadcast_order()     → sends to /ws/orders clients

    main.py           → FastAPI app entry point
                        - CORS middleware
                        - includes all routers
                        - /ws WebSocket endpoint
                        - /ws/orders WebSocket endpoint

3. AUTHENTICATION FLOW
----------------------------------------------------------------

REGISTER:
  POST /auth/register
  Body: { name, username, password, role, admin_key? }
  - If role=admin, admin_key must match ADMIN_REGISTRATION_KEY in .env
  - Password is hashed with bcrypt before storing
  - Returns: UserOut { id, name, username, role }

LOGIN:
  POST /auth/login
  Body: { username, password }
  - Verifies password hash
  - Creates JWT token with { sub: user_id, role, exp }
  - Token expires in ACCESS_TOKEN_EXPIRE_MINUTES (default 480 = 8 hours)
  - Returns: { access_token, token_type: "bearer", user: UserOut }

FRONTEND:
  - Token stored in localStorage["token"]
  - User object stored in localStorage["user"]
  - api.js interceptor auto-adds: Authorization: Bearer <token>
  - On 401 response → clears localStorage → redirects to /login

ROUTE GUARDS (App.jsx):
  - AdminGuard  → only role=admin, else redirect to /staff/tables
  - StaffGuard  → only role=staff, else redirect to /admin/dashboard
  - RoleRouter  → redirects based on role after login
  - /display/kitchen and /display/customer → NO auth guard (public)

4. ORDER LIFECYCLE
----------------------------------------------------------------

STEP 1 - SELECT TABLE (StaffDashboard)
  GET /tables → shows all tables with status
  Staff clicks an available table
  → table saved in OrderContext.selectedTable
  → navigate to /staff/order

STEP 2 - BUILD ORDER (StaffOrder)
  GET /products → shows product grid
  Staff adds items to cart (OrderContext.cart)
  Cart shows running total
  Staff clicks "Place Order"

STEP 3 - PLACE ORDER (StaffOrder → POST /orders/)
  Body: { table_id, items: [{ product_id, quantity }] }
  Backend (order_service.py):
    - Validates each product exists and is_available=true
    - Calculates total_amount
    - Inserts order into MongoDB with status="pending"
  Backend (orders.py):
    - Updates table status → "occupied"
    - Broadcasts to /ws:       { event: "new_order", order_id, table_id }
    - Broadcasts to /ws/orders: { type: "NEW_ORDER", data: <full order> }
  Frontend:
    - Clears cart
    - Navigates to /staff/payment

STEP 4 - KITCHEN PROCESSES ORDER
  Option A: StaffKitchen (/staff/kitchen)
    - Staff-facing kitchen view
    - Shows Pending and Preparing columns
    - Listens on /ws for real-time updates
    - Buttons: "▶ Start" (pending→preparing), "✓ Done" (preparing→completed)

  Option B: KitchenDisplay (/display/kitchen)
    - Dedicated full-screen KDS (meant for kitchen monitor)
    - Shows Pending, Preparing, Completed columns
    - Listens on /ws/orders for real-time updates
    - Same action buttons as StaffKitchen

  Status Update:
    PATCH /orders/{id}/status
    Body: { status: "preparing" | "completed" }
    Backend:
      - Updates order status in MongoDB
      - Broadcasts to /ws:       { event: "order_updated", order_id, status }
      - Broadcasts to /ws/orders: { type: "UPDATE_ORDER", data: <full order> }

STEP 5 - CUSTOMER SEES STATUS (CustomerDisplay)
  /display/customer → public URL, open on customer-facing screen/TV
  - Fetches today's active orders on load (GET /orders/)
  - Listens on /ws/orders for real-time updates
  - Shows order cards with table number, items, status badge
  - Status: ⏳ Pending → 👨‍🍳 Preparing → ✅ Ready
  - Paid orders automatically disappear from display

STEP 6 - PAYMENT (StaffPayment)
  GET /orders/ → shows orders with status pending/preparing/completed
  Staff selects order → chooses Cash or UPI
  POST /orders/{id}/pay
  Body: { payment_method: "cash" | "upi" }
  Backend:
    - Sets order status → "paid"
    - Sets payment_method
    - Updates table status → "available"
  Frontend:
    - Shows success animation
    - Refreshes order list

5. REAL-TIME WEBSOCKET SYSTEM
----------------------------------------------------------------

TWO WebSocket endpoints:

  /ws  (legacy, used by StaffKitchen)
    - Simple broadcast: { event: "new_order"|"order_updated", ... }
    - Triggers a re-fetch of /kitchen/orders on the client

  /ws/orders  (new, used by KitchenDisplay + CustomerDisplay)
    - Full order broadcast: { type: "NEW_ORDER"|"UPDATE_ORDER", data: <order> }
    - Client updates state directly without re-fetching

ConnectionManager (manager.py):
  - Maintains two separate connection lists
  - broadcast()       → sends to all /ws clients
  - broadcast_order() → sends to all /ws/orders clients
                        auto-removes dead connections

OrderSocket (socket.js) - Singleton:
  - One shared WebSocket instance for the whole app
  - Multiple components can subscribe/unsubscribe independently
  - Auto-reconnects every 3 seconds if connection drops
  - subscribe(fn) → returns unsubscribe function (used in useEffect cleanup)

When is broadcast triggered?
  POST /orders/       → NEW_ORDER broadcast to both /ws and /ws/orders
  PATCH /orders/status → UPDATE_ORDER broadcast to both /ws and /ws/orders
  POST /orders/pay    → NO broadcast (order leaves active display naturally)

6. ADMIN FEATURES
----------------------------------------------------------------

  DELETE /products/{id}  → delete product

AdminOrders:
  GET /orders/           → all orders with full details

AdminAnalytics:
  GET /analytics/overview      → totals (orders, revenue, tables, products)
  GET /analytics/daily-revenue → last 7 days revenue + order count per day
  GET /analytics/top-products  → top 6 products by quantity sold

KITCHEN
  GET    /kitchen/orders         → active orders (pending + preparing)

ANALYTICS
  GET    /analytics/overview     → full stats overview
  GET    /analytics/daily-revenue → 7-day revenue chart data

  GET    /analytics/staff         → staff list

WEBSOCKETS
  WS     /ws                     → real-time for StaffKitchen
  WS     /ws/orders              → real-time for KitchenDisplay + CustomerDisplay

10. HOW TO RUN
----------------------------------------------------------------

Backend:
  cd backend
  pip install -r requirements.txt
  uvicorn app.main:app --reload --port 8000

Frontend:
  cd frontend
  npm install
  npm run dev

Seed data (optional):
  python backend/seed.py        → seeds products + tables
  python backend/seed_admin.py  → creates default admin user

Access:
  App:              http://localhost:5173
  API docs:         http://localhost:8000/docs
  Kitchen Display:  http://localhost:5173/display/kitchen
  Customer Display: http://localhost:5173/display/customer
